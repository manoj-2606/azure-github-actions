PHASE 1 GOALS — Networking Foundation
------------------------------------------------------------
[ ] Create a Hub VNet (10.0.0.0/16) in Central India
[ ] Create a Spoke VNet (10.1.0.0/16) in Central India
[ ] Add subnets to Spoke VNet:
      - app-subnet     (10.1.1.0/24)
      - data-subnet    (10.1.2.0/24)
      - pe-subnet      (10.1.3.0/24)  ← private endpoints go here
[ ] Add hub-subnet to Hub VNet (10.0.1.0/24)
[ ] Configure VNet Peering: Hub → Spoke
[ ] Configure VNet Peering: Spoke → Hub (peering must be bidirectional)
[ ] Understand peering options:
      - allow_forwarded_traffic = true
      - allow_gateway_transit   = false (no gateway today)
      - use_remote_gateways     = false (no gateway today)

------------------------------------------------------------
PHASE 2 GOALS — Private Endpoints
------------------------------------------------------------
[ ] Deploy Azure Storage Account with public_network_access = "Disabled"
[ ] Deploy Azure Key Vault with public_network_access = "Disabled"
[ ] Create Private Endpoint for Storage Account in pe-subnet
[ ] Create Private Endpoint for Key Vault in pe-subnet
[ ] Create Private DNS Zone for Storage: privatelink.blob.core.windows.net
[ ] Create Private DNS Zone for Key Vault: privatelink.vaultcore.azure.net
[ ] Link both Private DNS Zones to Spoke VNet
[ ] Link both Private DNS Zones to Hub VNet
[ ] Understand: without DNS zone linking, private endpoint won't resolve

------------------------------------------------------------
PHASE 3 GOALS — NSG Hardening
------------------------------------------------------------
[ ] Create NSG and associate with app-subnet
[ ] Create NSG and associate with pe-subnet
[ ] Apply inbound rule: allow traffic from app-subnet to pe-subnet (port 443)
[ ] Apply inbound rule: deny all other inbound
[ ] Apply outbound rule: allow only necessary outbound
[ ] Understand: NSG = firewall at subnet level, not VNet level

LEARNING OBJECTIVES
------------------------------------------------------------
After completing Day 12, you should be able to explain:

1. WHY Hub-and-Spoke?
   → Centralised control, shared services in hub, workloads isolated in spokes

2. WHY Private Endpoints?
   → Service gets a private IP inside your VNet — no traffic leaves Azure backbone
   → Public endpoint is disabled — attackers cannot reach the service from internet

3. WHY Private DNS Zone?
   → When your VM resolves "mystorageaccount.blob.core.windows.net",
     it must get the PRIVATE IP back (10.1.3.x), not the public IP
   → Without DNS zone linked to VNet, it resolves to public IP — private endpoint useless

4. WHY NSG hardening?
   → VNet peering allows all traffic by default
   → NSG adds subnet-level firewall rules — least privilege

5. WHY disable public network access?
   → GDPR / EU compliance requirement
   → Even if attacker gets credentials, they cannot reach the service from internet

------------------------------------------------------------
REAL-WORLD RELEVANCE (Finland / EU)
------------------------------------------------------------
Finnish and broader EU cloud environments routinely require:
  - No public blob storage endpoints (PCI-DSS, GDPR)
  - Private Key Vault access only (secrets never exposed publicly)
  - Network segmentation (SOC2, ISO27001 requirements)
  - Hub-Spoke topology for centralised firewall/NVA control
  - Audit logs for all network access attempts

If you can confidently build this, you are no longer junior-level.
You are cloud infrastructure capable.
