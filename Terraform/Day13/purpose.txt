DAY 13 — PURPOSE OF THIS PROJECT
==================================

THE CORE PROBLEM THIS SOLVES
------------------------------
You deployed infrastructure for 12 days.
It runs. It works. But you are completely blind.

If someone deletes a secret from Key Vault — you won't know.
If someone accesses your storage account from an unknown IP — you won't know.
If your NSG silently drops critical traffic — you won't know.
If an engineer makes a destructive change at 2am — you won't know.

That is not infrastructure. That is a liability.

This project adds eyes to your infrastructure.

WHAT LOG ANALYTICS WORKSPACE IS
---------------------------------
It is a centralized store for all your platform telemetry.
Think of it as a database that receives logs from every Azure resource.
It has its own query language: KQL (Kusto Query Language).
You send data INTO it. You query OUT of it.

It is NOT a dashboard (that is Azure Monitor Workbooks or Grafana).
It is NOT alerting (that is Azure Monitor Alerts on top of it).
It IS the single source of truth for what happened in your infrastructure.

WHAT DIAGNOSTIC SETTINGS ARE
------------------------------
Every Azure resource generates events internally.
By default, those events go nowhere. They are discarded.
Diagnostic Settings is a connector that says:
  "Take the events from THIS resource and send them TO this destination."

The destination can be:
- Log Analytics Workspace (querying, long-term analysis)
- Storage Account (archiving, compliance)
- Event Hub (real-time streaming to SIEM tools)

In this project you send everything to Log Analytics.

WHAT ACTIVITY LOGS ARE
-----------------------
Activity Logs are NOT resource-level logs.
They are subscription-level operational logs.
They record: who did what, when, from where, and with what result.
Examples:
  - "User X deleted a resource group at 03:14"
  - "Service principal Y created a VM"
  - "Role assignment changed on Key Vault"

These are the logs that matter for:
  - Security incident investigation
  - Compliance audits (GDPR, ISO 27001, SOC2)
  - Change tracking

WHY CENTRALIZATION MATTERS
----------------------------
If each team sends logs to their own workspace, you have:
  - No cross-resource correlation
  - No unified security view
  - Duplicate costs
  - Compliance gaps

Centralized = one workspace, all resources, one query surface.
This is the pattern used in enterprise Azure Landing Zones.

WHY THIS MATTERS FOR A JOB IN FINLAND/EU
------------------------------------------
GDPR Article 32 requires "appropriate technical measures" for data security.
NIS2 Directive requires monitoring of critical infrastructure.
ISO 27001 (common in Finnish tech companies) requires audit logging.

When an interviewer asks: "How do you handle compliance in your infra?"
This is your answer. Not theory. You built it.
